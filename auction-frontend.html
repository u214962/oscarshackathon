Hash function design
Single-layer keccak256 with two parameters

bidHash = keccak256(abi.encodePacked(amount, password))

The two inputs serve distinct purposes:
  amount — the actual bid value (hidden during bidding)
  password — a secret salt that prevents brute-force

Security properties:
  Pre-image resistance: given only bidHash, nobody can recover amount or password
  Hiding: bid value is invisible on-chain during the bidding phase
  Binding: after committing the hash, the bidder cannot change amount or password
  Front-running resistance: miners see only the hash, not the value

Known limitation:
If the amount range is small and known (e.g. 1-100 ETH), an attacker could try all amounts. The password prevents this: brute-force requires iterating all (amount, password) pairs, which is infeasible with a strong password.
Full protocol flow
Phase BIDDING
Bidder computes bidHash = keccak256(amount, password) off-chain.
Calls bid(bidHash) with the required deposit. Nothing is visible.

Phase REVEAL
Bidder calls reveal(amount, password). Contract checks:
  keccak256(amount, password) == stored bidHash
Match: bid accepted. No match: deposit forfeited.

Phase CLAIM
Winner calls claimPrize(amount, password). Same hash check as reveal, acts as a second authentication factor. If winner doesn't claim, skipClaimant() advances to the next highest bidder.

Phase ENDED
All bidders who revealed correctly call refund() to recover deposit. Organizer calls withdrawForfeited() for forfeited deposits.
